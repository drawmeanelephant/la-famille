# fix/rag-delimiter-escaping

## Problem

The documented `ask` quickstart (`content/docs/ask.md`) fails on this repo:

```bash
go run ./cmd/la-famille rag     # OK
go run ./cmd/la-famille ask     # Error
# ask: prepare corpus: retrieval: parse rag-system.md: <content> outside of <file>
```

The RAG archive wraps each file verbatim in `<file path="...">` / `<content>`
markers, and `internal/retrieval/loader.go` parses them with a line-prefix state
machine. Content lines are written unescaped, so any bundled file whose body
contains those markers at the start of a line injects real structure.

`rag-system.md` bundles the project's own Go sources, and `rag-content.md`
bundles every page under `content/`. So the archive poisons itself:
`internal/retrieval/chunker_test.go` holds raw-string fixtures containing
`<file path="...">`, `<content>`, `</content>` and `</file>`. Marker counts in a
freshly generated `rag-system.md` were 128 `<file path=` against 121 `</file>`.

Causation was confirmed by removing that one test file: `ask` then started and
`/api/status` returned 200; restoring it reproduced the failure.

This is not limited to test fixtures. Any user Markdown page documenting the
RAG format breaks that user's own `ask`.

## Decision

Escape at **write** time, unescape at **read** time.

Rejected: making the parser tolerant by tracking state. The fixture in
`chunker_test.go` contains `</content>` immediately followed by `</file>` — a
correctly-formed terminator sequence — so no amount of state tracking can
distinguish it from real structure. A lenient parser would silently mis-chunk
instead of failing loudly, which is worse than the current error.

Rejected: a length-prefixed or nonce-delimited format. Larger change, and it
would hurt the archive's readability for the external LLM consumers it exists
to serve.

## Design

New `internal/ragfmt` owns the single escaping rule, imported by both the writer
(`internal/ragexport`) and the reader (`internal/retrieval`), so the two can
never drift. No import cycle: neither package previously imported the other.

Rule: a content line whose first characters are a marker — optionally already
preceded by backslashes — gains one leading backslash. The reader removes
exactly one. Reversible for arbitrary input; a line that merely starts with a
backslash (`\hello`) is untouched, so ordinary prose and code are unaffected.
CRLF endings are preserved.

## Files

- `internal/ragfmt/ragfmt.go` (new), `internal/ragfmt/ragfmt_test.go` (new)
- `internal/ragexport/export.go` — escape bodies in `writeBundle`
- `internal/retrieval/loader.go` — unescape body lines
- `internal/retrieval/loader_escaping_test.go` (new) — end-to-end regression

## Breaking changes to static asset generation

None. The site build (`la-famille build`) is untouched. `rag-archive/` is
gitignored and regenerated by `la-famille rag`, so there is no committed corpus
to migrate. Archives produced by an older writer that contain unescaped markers
still fail loudly rather than mis-parsing — covered by a test.

## Validation

- `go test ./...`
- `go vet ./...`
- Manual: `la-famille rag` then `la-famille ask` with `chunker_test.go` present
  — server starts, `/api/status` returns 200.
